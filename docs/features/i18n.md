# Internationalization (i18n)

> **Supported user locales:** English (`en`) and Brazilian Portuguese (`pt-BR`) only.  
> **Coverage rule:** Every user-visible string MUST go through i18n — no hardcoded UI copy in components or API error titles shown to users.

## Policy split (non-negotiable)

| Layer | Language | Rule |
|-------|----------|------|
| **User-facing UI** | `en` + `pt-BR` | 100% i18n — labels, buttons, errors, empty states, meta titles |
| **User-facing API text** | `en` + `pt-BR` | RFC 7807 `title` / `detail` resolved from locale |
| **Catalog content** | `en` + `pt-BR` | Product/category name and descriptions stored per locale (see DB) |
| **WhatsApp order message** | `pt-BR` always | Brazilian channel — independent of UI locale ([whatsapp-flow.md](whatsapp-flow.md)) |
| **Code, identifiers, logs** | English | Harness `english-only.mdc` — unchanged |
| **Repo docs, commits** | English | Harness — unchanged |

## Locale identifiers

| Code | UI label | Usage |
|------|----------|-------|
| `en` | English | Default fallback when translation missing |
| `pt-BR` | Português (Brasil) | Default for first visit when `Accept-Language` prefers Portuguese |

No other locales (e.g. `es`, `pt-PT`) — reject or fall back to `en`.

## Locale negotiation

### Frontend (`apps/web`)

| Priority | Source |
|----------|--------|
| 1 | User choice (language switcher) → persisted `localStorage` key `locale` **and** cookie `locale` |
| 2 | URL prefix `/en/...` or `/pt/...` (TanStack Router) |
| 3 | Browser `Accept-Language` (SSR) / `navigator.language` (client bootstrap) |
| 4 | Default `pt-BR` |

SSR uses the same locale for React Query loaders and i18n: request middleware resolves cookie → `Accept-Language` → default, then dehydrates the query cache via `@tanstack/react-router-ssr-query` so the client hydrates without markup mismatches.

API requests from the web app MUST send:

```
Accept-Language: en
```

or

```
Accept-Language: pt-BR
```

### API (`apps/api`)

| Input | Resolution |
|-------|------------|
| `Accept-Language` header | Primary |
| `?locale=en` or `?locale=pt-BR` | Override (optional, for debugging) |
| Invalid / missing | `pt-BR` for catalog in production; `en` as secondary fallback chain |

Resolved locale is attached to request context and passed to use cases.

## Stack (decided)

| Layer | Library / pattern |
|-------|-------------------|
| Web UI | [i18next](https://www.i18next.com/) + [react-i18next](https://react.i18next.com/) |
| Web messages | JSON files under `apps/web/src/i18n/locales/` |
| Shared locale type | `@print3d/shared-types` — `SupportedLocale`, `DEFAULT_LOCALE` |
| API errors | `apps/api/src/i18n/` — keyed messages for RFC 7807 |
| DB catalog text | JSONB `translations` column per entity (see below) |
| Full-text search | Separate `search_vector_en` / `search_vector_pt` (Phase 4 migration) |

## Database — translatable catalog fields

Products and categories expose **locale-specific** text to the API. Storage pattern:

```json
{
  "en": {
    "name": "Custom Photo Frame",
    "description": "Full description…",
    "shortDescription": "Photo frame with embossed name"
  },
  "pt-BR": {
    "name": "Porta-retrato personalizado",
    "description": "Descrição completa…",
    "shortDescription": "Porta-retrato com nome em relevo"
  }
}
```

| Column | Type | Notes |
|--------|------|-------|
| `translations` | `jsonb` | Keys: `en`, `pt-BR` only |
| `search_vector_en` | `tsvector` | Trigger: `to_tsvector('english', …)` |
| `search_vector_pt` | `tsvector` | Trigger: `to_tsvector('portuguese', …)` |

Slug remains **single** per product (URL-stable, not translated).

Migration: Phase 4 — Task 4.10.

## API response shape (locale-aware)

`GET /products/:slug` with `Accept-Language: pt-BR` returns resolved fields:

```json
{
  "data": {
    "id": "01935…",
    "slug": "custom-photo-frame",
    "name": "Porta-retrato personalizado",
    "shortDescription": "Porta-retrato com nome em relevo",
    "locale": "pt-BR"
  }
}
```

English request returns the `en` branch. Contract details: [../api/contract.md](../api/contract.md#internationalization).
